### "How to run a business from one laptop as a college student"

**Target length:** 1,000-1,200 words
**Product tie-in:** Solopreneur OS ($12.99) + Behike Method v2 ($19.99) — behike.shop

I have four hours a week of focused build time, not counting classes. Twenty-five products shipped in six weeks.

The time is not the constraint. The system is.

---

**[INTRO — 0:15-0:45]**

[B-roll: class schedule, then time blocks marked for building]

I'm a computer engineering student. The courseload is real. The projects are real. I don't have eight hours a day to run a business.

What I have is 45-minute blocks, usually in the early morning or late at night, where I can work without interruption. Everything I've built happened in those blocks.

The system I built around those constraints is what made it possible.

---

**[SECTION 1 — 0:45-2:30]**
**[00:45] Constraint-driven design**

[B-roll: calendar with class blocks, work blocks, and rest blocks color coded]

When you design a system around unlimited time, you get a system that requires unlimited time.

I designed mine around 45 minutes. Every component of the business has to work within that unit.

A 45-minute session can: produce one complete product section, write and schedule one week of content, build and test one automation workflow, publish two products on Gumroad.

[B-roll: timer running, task completed within the block]

The limit creates clarity. I'm never asking "what should I work on today." The session structure tells me.

---

**[SECTION 2 — 2:30-4:15]**
**[02:30] The session structure**

[B-roll: primer.md file, session start ritual visible]

Every build session starts the same way. Five minutes: read primer.md. Understand the current state. Identify the one thing to finish this session.

Then 35 minutes of uninterrupted work on that one thing.

Then five minutes: update primer.md with what was done and what comes next.

[B-roll: timer showing 5-35-5 breakdown]

Forty-five minutes. No exceptions to the format. The ritual is what makes the work feel continuous even when the sessions are days apart.

---

**[SECTION 3 — 4:15-6:00]**
**[04:15] Background agents as time multiplication**

[B-roll: background agent task queue, tasks running while laptop lid is down]

The most important thing I do is start background agent tasks at the end of every active session.

I finish my 45 minutes. I write a task brief for the next thing. I start the background agent. I close the laptop.

By the time I come back, that task is done. My next session starts with new material, not an empty screen.

[B-roll: session ending, task brief written, agent started, notification received later]

This is how six weeks produced 25 products. Active sessions do the high-judgment work. Background agents handle everything else.

---

**[SECTION 4 — 6:00-7:45]**
**[06:00] Batching to reduce context switching**

[B-roll: single-category work session, all similar tasks grouped]

Context switching is expensive. Finishing a coding task and immediately shifting to writing a product listing costs 15 minutes of reset time each direction.

I batch by type. Monday evening: writing and content. Tuesday morning: code. Wednesday: publishing and Gumroad. Thursday: research.

[B-roll: weekly batch schedule in a simple table]

When I sit down to write, I only write. When I sit down to code, I only code. The focused output per hour is dramatically higher than switching between types every session.

---

**[SECTION 5 — 7:45-9:30]**
**[07:45] The tools that work for this constraint**

[B-roll: minimal tool stack visible on screen, five items]

Most productivity tools assume you have time to configure them. I don't.

My rule: if setup takes longer than one session, it's too heavy.

Claude, VS Code, n8n, CapCut, and flat files. All of them were set up in under an hour each. All of them still work the same way months later.

[B-roll: each tool opening quickly, no complex setup visible]

Reliability matters more than features when your sessions are short. A tool that requires maintenance is a tool that will break at the wrong moment.

---

**[SECTION 6 — 9:30-10:45]**
**[09:30] What makes it sustainable**

[B-roll: two-week sprint calendar, followed by a rest day marked clearly]

I treat the business like a class. It has scheduled time. When that time is up, I stop.

I don't optimize for more hours. I optimize for better hours.

The reason this works long-term: I don't resent the work. I look forward to the sessions because they're contained. I know they end. I know what to do when they start.

[B-roll: primer.md with a clear "next action" that requires no thinking to start]

The session structure removes decision fatigue at the start. The 45-minute limit prevents burnout at the end. Both of those are system design choices, not willpower.

## SCRIPT 3
### "Content batching: how I make 30 posts in one afternoon"

**Target length:** 1,000-1,200 words
**Product tie-in:** Social Media Pipeline Guide ($4.99) + Content Waterfall ($4.99) — behike.shop
**Thumbnail concept:** Stack of content pieces. Post counter visible: 1... 10... 30. Text: "One afternoon."

---

**[HOOK — 0:00-0:15]**

[B-roll: file folder opening, 30 Instagram caption files visible]

Thirty posts. One afternoon. Most of them took less than five minutes each.

Here's the system.

---

**[INTRO — 0:15-0:45]**

[B-roll: laptop, note with content categories written]

Content batching is not about working faster. It's about working in the right mode.

When you're in writing mode, you write. You don't switch to editing, scheduling, or research. You stay in the same cognitive state for two hours and produce as much as that state allows.

That's it. The whole system follows from that principle.

---

**[SECTION 1 — 0:45-2:30]**
**[00:45] The category system**

[B-roll: content category list: insight, product, personal, educational, CTA]

Every piece of content I produce fits into one of five categories. Insight: a specific thing I observed or learned. Product: something I built, with context. Personal: a story or reflection. Educational: a step-by-step or how-to. CTA: direct invitation to buy or follow.

[B-roll: the five categories labeled, one example under each]

Before a batching session, I decide the category mix. Typically: twelve insights, eight educational, five personal, three product, two CTA. Thirty total.

The categories tell me what to write before I open a blank page. No blank page paralysis.

---

**[SECTION 2 — 2:30-4:15]**
**[02:30] The brief-first approach**

[B-roll: brief file for 30 posts, one line per post, category and topic noted]

Before I write a single caption, I write briefs. One line per post. Category, topic, one-sentence angle.

"Insight. ADHD and building. The thing about hyperfocus is that it's not discipline. It's interest."

That brief takes 30 seconds. Writing from a brief takes three minutes. Writing from nothing takes fifteen.

[B-roll: brief file being populated, then used as reference while writing]

Thirty briefs take about 15 minutes. Then the writing session is fast because every blank post already has a direction.

---

**[SECTION 3 — 4:15-6:00]**
**[04:15] Using AI to accelerate, not replace**

[B-roll: brief pasted into Claude, output appearing, editing immediately after]

I paste my briefs into Claude ten at a time. "Write ten Instagram captions using these briefs. Journalism structure. Short sentences. No exclamation marks. No em dashes. Lead with the observation, not with I."

Claude produces ten drafts. I read each one and make three to five word changes per caption.

[B-roll: specific word changes in real time, before and after on screen]

The AI doesn't write my content. It writes the frame. My edit pass puts the voice in. That distinction is why the output sounds like me, not like a bot.

---

**[SECTION 4 — 6:00-7:45]**
**[06:00] The editing pass**

[B-roll: draft captions in a document, tracked changes visible]

Editing in batches is faster than editing one by one. My brain is in critic mode for the whole pass, not switching between writer and critic.

What I look for in the edit:

Does the first line stop the scroll. Is there a specific detail or number. Does it sound like something I'd actually say. Is the last line a forward-looking statement, not a summary.

[B-roll: caption before and after edit, specific improvements visible]

Most edits are small. One word replaced. One sentence cut. One reordering. The brief was good. The AI frame was close. The edit just makes it real.

---

**[SECTION 5 — 7:45-9:30]**
**[07:45] Organizing the output**

[B-roll: content folder with files named by category and date]

After the editing pass, I save everything to a structured folder. Category, batch number, date.

I don't schedule directly from the session. I schedule separately. Mixing production and scheduling extends the session and exhausts the wrong kind of mental energy.

[B-roll: content folder, then scheduling tool opened separately later]

Production and publishing are two different jobs. Keep them in different sessions.

---

**[SECTION 6 — 9:30-10:45]**
**[09:30] The timeline: what one afternoon actually looks like**

[B-roll: timer running through each phase]

Here's the actual time breakdown for 30 posts.

Fifteen minutes: write all 30 briefs.
Thirty minutes: run briefs through Claude in three batches of ten.
Forty-five minutes: edit pass for all 30.
Fifteen minutes: organize output.

Total: one hour and forty-five minutes. Call it two hours with breaks.

[B-roll: final folder of 30 captions, timestamp visible]

Two hours. One month of daily Instagram content. That's not fast. That's just batching and staying in the right mode.

## SCRIPT 4
### "The morning briefing system that keeps me building consistently"

**Target length:** 1,000-1,200 words
**Product tie-in:** Solopreneur OS ($12.99) — behike.shop
**Thumbnail concept:** Morning desk. Coffee. Laptop opening. Clock showing 7am. Text: "The 10-minute ritual."

---

**[HOOK — 0:00-0:15]**

[B-roll: laptop opening, primer.md loading immediately, no browser tab visible]

Ten minutes. Every morning. This is the only habit that kept me building consistently for six weeks.

---

**[INTRO — 0:15-0:45]**

[B-roll: morning desk setup, minimal, one window visible]

Consistency is not a personality trait. It's a system outcome.

I'm not naturally disciplined. I have ADHD. I lose context easily. I start a lot of things. Without a system that holds context for me, projects drift.

The morning briefing is the system that holds context. Here's how it works.

---

**[SECTION 1 — 0:45-2:30]**
**[00:45] The problem it solves**

[B-roll: blank laptop screen, cursor waiting, no clear starting point]

The most common reason I'd lose momentum was the "where was I" problem. I'd sit down to work and spend 15 minutes reconstructing context before I could do anything real.

At 45 minutes per session, that's a third of the session gone.

[B-roll: the same session with primer.md already loaded, starting immediately]

The morning briefing eliminates that. I open the laptop and the first thing I read tells me exactly where I am, what was last completed, and what the next action is.

No reconstruction. Zero minutes lost.

---

**[SECTION 2 — 2:30-4:15]**
**[02:30] What the briefing contains**

[B-roll: primer.md file, scrolling through sections]

My morning briefing is a file called primer.md. It has five sections.

Current project: what I'm building right now, one paragraph.
Last session: what was completed in the previous session, bullet points.
Next action: the specific thing to do today, one sentence.
Blockers: anything that requires a decision or external input before I can continue.
Background agents: what's running, what's completed, what needs review.

[B-roll: each section highlighted as it's named]

Five sections. The whole file is usually 30-50 lines. I read it in two minutes.

---

**[SECTION 3 — 4:15-6:00]**
**[04:15] The ritual**

[B-roll: morning session opening: primer.md, then one task, then work]

The ritual is three steps.

Read primer.md. Understand the current state.

Open the one file related to the next action. Not five files. One.

Work for the session.

[B-roll: each step taking under two minutes, then 35 minutes of uninterrupted work]

The ritual is fast because the decisions were already made when primer.md was last updated. I'm not planning in the morning. I'm executing the plan I made yesterday.

---

**[SECTION 4 — 6:00-7:45]**
**[06:00] Updating the file: the end-of-session protocol**

[B-roll: session ending, primer.md being updated]

The morning briefing only works if it's updated at the end of every session. That's the maintenance cost.

End-of-session update takes five minutes. I rewrite "last session" with what just happened. I update "next action" with the specific thing to do next. I update "background agents" with any new tasks started.

[B-roll: cursor moving through each section, quick updates]

Five minutes at the end of a session saves fifteen at the start of the next one. The math is easy.

---

**[SECTION 5 — 7:45-9:30]**
**[07:45] What happens when you miss a session**

[B-roll: calendar with a few missed days marked, then a return session]

Some weeks I miss two or three days. Work, coursework, life.

When I come back, primer.md is still there. It doesn't decay. It holds the state exactly as I left it.

[B-roll: return session, primer.md loading cleanly, no momentum lost]

This is what I mean by the system holding context for me. I can't trust my memory across a three-day break. I can trust the file.

The system compensates for the ADHD. Not by fighting it. By building something that works around it.

---

**[SECTION 6 — 9:30-10:45]**
**[09:30] The psychological effect**

[B-roll: comparison: starting with no system vs. starting with primer.md]

There's a motivation component here that's hard to measure.

When I sit down and immediately know what to work on, I feel competent. That feeling helps me start. Starting is where most of the work lives.

When I sit down and have to reconstruct everything, I feel behind before I've done anything. That feeling is demotivating. It makes the session shorter than it needs to be.

[B-roll: session timer starting immediately vs. starting after 15-minute delay]

The file is not just a productivity tool. It's a confidence tool. It tells you: you were here, you made progress, here's the next step.

## SCRIPT 5
### "Primer.md: the file that gives me context every single session"

**Target length:** 1,000-1,200 words
**Product tie-in:** Solopreneur OS ($12.99) — behike.shop
**Thumbnail concept:** Terminal open, primer.md loading. Text: "The file everything connects to."

---

**[HOOK — 0:00-0:15]**

[B-roll: terminal command: cat primer.md, content appearing]

This file is 50 lines. It runs my entire business.

Not metaphorically. Literally.

---

**[INTRO — 0:15-0:45]**

[B-roll: primer.md full view, section headers visible]

primer.md is not a productivity hack. It's a memory system.

I have ADHD. Context doesn't persist naturally from session to session. Projects drift. Priorities shift. Work gets lost.

primer.md is the thing I built to solve that. Not an app. Not a database. A plain text file that I rewrite at the end of every session so that the next session can start without rebuilding from scratch.

This is how it works and why it matters more than any tool in my stack.

---

**[SECTION 1 — 0:45-2:30]**
**[00:45] The single source of truth principle**

[B-roll: multiple apps, multiple tools, then a single file overriding all of them]

The problem with most productivity systems: they spread state across too many places.

Task in Notion. Note in Obsidian. Thread in email. Voice memo on phone. Slack message. Calendar block.

When you need to know what's happening right now, you check five places. That friction adds up to real time and real mental load.

[B-roll: all those tools replaced by a single file open on screen]

primer.md is the single source of truth for my current state. Not the whole project history. Not my long-term goals. Just: what am I building right now, what did I just do, and what's next.

---

**[SECTION 2 — 2:30-4:15]**
**[02:30] The structure in detail**

[B-roll: primer.md open, each section zoomed in as it's described]

The file has six sections. I'll go through each one.

Live state: a two-paragraph description of the current project and phase. What I'm building, what milestone I'm approaching.

Last completed: bullet points from the previous session. What was done, with file names if relevant.

Next action: one sentence. The specific thing to do in the next session. Not "work on product" but "finish section 3 of the AI security guide and export to PDF."

Blockers: anything that needs a decision or external action before I can continue. Gumroad setup. Domain purchase. API key.

Background agents: list of running or recently completed tasks. Status for each.

Open questions: things I noticed but didn't address. Ideas I don't want to lose. Problems I noticed.

---

**[SECTION 3 — 4:15-6:00]**
**[04:15] How Ceiba reads it**

[B-roll: primer.md being loaded into a Claude session, context-aware response following]

This file is not just for me. I load it at the start of every Claude session.

Because Claude has read primer.md, it already knows the current state. I don't have to re-explain what I'm building. I don't have to summarize the last three sessions.

[B-roll: Claude session starting with primer.md context, first response showing immediate project awareness]

The collaboration starts immediately. That's not possible if you open a blank chat window and start typing from scratch.

primer.md is the bridge between sessions. For me and for the AI I work with.

---

**[SECTION 4 — 6:00-7:45]**
**[06:00] The end-of-session update ritual**

[B-roll: session ending, primer.md being rewritten]

The file is only useful if it's current. That means rewriting it at the end of every session.

Not appending. Rewriting.

The rewrite takes five minutes. I move "next action" to "last completed." I write a new "next action" based on where I stopped. I update background agent status. I clear anything from "open questions" that got resolved.

[B-roll: clean rewrite, new version saved, clear sections]

Rewriting instead of appending keeps the file short. Short files get read. Long files get skimmed.

---

**[SECTION 5 — 7:45-9:30]**
**[07:45] The difference it makes in practice**

[B-roll: two session starts side by side: with primer.md vs. without]

I tracked this informally. Sessions that started by reading primer.md reached full focus in under three minutes. Sessions that started without it took twelve to fifteen minutes to orient.

For a 45-minute session, that difference is the session.

[B-roll: session timer graph showing early vs. late focus point]

It also reduced the chance of working on the wrong thing. "Next action" tells me what's most important. Without it, I'd default to whatever felt easiest, which is rarely the highest value thing.

---

**[SECTION 6 — 9:30-10:45]**
**[09:30] When the file fails you (and how to recover)**

[B-roll: outdated primer.md, last updated three weeks ago]

The failure mode is simple: you skip the end-of-session update.

One skip is fine. Two skips means the file is outdated. Three skips and it's no longer the source of truth.

[B-roll: recovery session: reading old commits, file names, recent exports]

Recovery: spend 15 minutes reconstructing the current state from file timestamps, recent git commits, and whatever's in your export folder. Then rewrite primer.md from that reconstruction. Then commit to the habit again.

The reconstruction is annoying enough that it usually resets the habit. You don't want to do it twice.
