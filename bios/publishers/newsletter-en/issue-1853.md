# Issue 1853: The Second Problem
**One insight. No fluff.**

When something goes wrong, most people solve the first problem. The bug gets fixed. The argument gets resolved. The fire gets put out. Then they move on and the second problem stays invisible.

The second problem is why the first one happened. The bug exists because the review process was skipped. The argument happened because a boundary was not set earlier. The fire started because a warning was ignored three weeks ago. First problems are symptoms. Second problems are causes.

Solving only symptoms is expensive. They return, usually in a slightly different form. Solving causes is slower upfront and much cheaper over time. It requires sitting with a problem longer than feels necessary and asking "what made this possible?" instead of "how do I make this stop?" That second question is where real improvement lives.
